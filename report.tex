\section{Introduction}
The Edge Fire Detection system generates real-time alerts based on neural network inference operating on edge cameras. To prevent notification fatigue during a sustained fire event, while simultaneously retaining a real-time visual representation of the fire's severity, a two-stage dynamic alert mechanism was implemented. This report outlines the architectural design of this mechanism.

\section{Mechanism Design}

The alerting mechanism is split into Edge Network Throttling and Cloud Event Aggregation.

\subsection{Edge Network Throttling}
At the edge node (e.g., Binh Phuoc), the PyTorch inference engine processes frames sequentially. To prevent flooding the MQTT broker with an alert payload for every frame (e.g., 30 payloads per second), a local camera-level throttle is applied. The inference engine guarantees a maximum transmission frequency of exactly one payload per camera every 10 seconds. 

\subsection{Cloud Event Aggregation and Notification Suppression}
The Cloud-native Exporter subscribes to the MQTT broker and maps incoming payloads to the Prometheus telemetry stack. When a fire payload arrives:
\begin{enumerate}
    \item \textbf{Initial Alert}: The Exporter logs an active alert, dispatches a Telegram notification containing the captured frame, and plots a pinpoint on the Grafana Geomap.
    \item \textbf{Cooldown Window}: The Exporter enters a strictly enforced 120-second (2-minute) cooldown window for that specific camera location. Any subsequent fire payloads received from this camera within this window will \textbf{not} trigger a Telegram dispatch.
\end{enumerate}

\section{Dynamic Visual Emphasis (Grafana Integration)}
Although Telegram notifications are suppressed during the 120-second window, the telemetry data is not discarded. Instead, the Exporter intercepts the silent payloads and increments an internal tracking gauge (\texttt{fire\_frames\_count}).

This metric is exported to Prometheus and queried by the Grafana Geomap dashboard. The size of the red indicator dot on the Geomap is dynamically bound to this count metric. Consequently:
\begin{itemize}
    \item A brief, unconfirmed detection will result in a standard, static red dot.
    \item A sustained, raging fire will cause the red dot to geometrically expand over time as the \texttt{fire\_frames\_count} increases.
\end{itemize}

This provides operators with an immediate, intuitive visual measure of the fire's severity directly on the monitoring dashboard without experiencing notification spam.
